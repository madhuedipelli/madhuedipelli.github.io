    The 7 System Design Gotchas to Avoid

    Based on comprehensive analysis of "You Suck At Coding Interviews" guide.

    1Ô∏è‚É£ Premature Component Specification ‚≠ê MOST COMMON

    Failure Pattern:

    Interviewer: "How would you scale Twitter?"
You: "We'll use MongoDB, Redis, Nginx, Kafka, Cassandra..."
Interviewer: "Why MongoDB?"
You: "Because... it scales?"
Interviewer: ‚ùå FAIL

    Root Cause: Tech stack memorization, not reasoning

    Why It Happens: - You've heard MongoDB used at scale - Don't understand WHY it was chosen - Can't defend your choice when questioned

    Fix: Start with PROBLEM characteristics, then JUSTIFY tech choice

    Correct Pattern:

    ‚ùå "We need MongoDB"

‚úÖ "We need to store non-relational data with horizontal scaling.
   MongoDB provides this by sacrificing consistency for availability.
   Trade-off: eventual consistency acceptable for tweets,
   critical for follower list (use relational here)"

    The Framework: 1. Problem: Define constraints (DAU, QPS, storage) 2. Need: What characteristic solves this? (consistency? horizontal scale? fast reads?) 3. Technology: Which tech provides this? (justify) 4. Trade-off: What are we sacrificing? (name it)

    Example:

    Problem: 500M tweets/month, 100K QPS, read-heavy (100:1 ratio)
Need: Horizontal scaling, fast reads, eventual consistency OK
Technology: Cassandra (distributed, read-optimized)
Trade-off: No powerful queries, eventual consistency

NOT "Cassandra is cool" - that fails immediately

    2Ô∏è‚É£ The Free Lunch Fallacy

    Myth: "NoSQL scales better than SQL"

    Truth: NoSQL trades consistency/joins for scale - no magic

    Reality Check:

    Aspect MySQL NoSQL
    Joins ‚úÖ Fast, powerful ‚ùå Complex, app-level
    horizontal scale ‚ö†Ô∏è Hard (sharding) ‚úÖ Built-in
    Consistency ‚úÖ Strong ‚ùå Eventual
    ACID ‚úÖ Yes ‚ùå Varies
    Simplicity ‚úÖ SQL predictable ‚ùå Schema-less surprises

    Your Takeaway: EVERY choice costs something. Name the cost.

    Interview Example:

    You: "Let's use NoSQL for everything, it scales better"
Interviewer: "What if we need to transfer money between accounts?"
You: "...that needs ACID transactions"
Interviewer: "NoSQL ‚úó ACID ‚úó Trade-off ‚úó"

Better:
You: "NoSQL for user timeline (eventual consistency OK)
SQL for transactions (need ACID)
Reason: NoSQL scales horizontally, SQL has strong guarantees"

    Common Free Lunch Beliefs: - ‚ùå "Cache makes everything fast" ‚Üí Cache misses, invalidation complexity - ‚ùå "Microservices solve scalability" ‚Üí Network latency, distributed tracing - ‚ùå "Multiple databases = best solution" ‚Üí Operational complexity, debugging nightmares

    3Ô∏è‚É£ Not Calculating Capacity ‚≠ê CAUSES WRONG DESIGN

    What Fails: Design without numbers

    Example:

    Interviewer: "Design YouTube"
You: "Multiple servers, load balancer, cache, database"
Interviewer: "How many servers?"
You: "Uh... as many as needed?"
Interviewer: ‚ùå FAIL

    Why Numbers Matter: Wrong capacity ‚Üí completely different architecture

    Example Calculation:

    500M URLs created/month
√∑ 30 days √∑ 86,400 seconds = 200 writes/sec

100:1 read/write ratio = 20,000 reads/sec

Each URL entry: 500 bytes average

3 years retention:
  500M URLs/month √ó 36 months = 18B total URLs
  18B URLs √ó 500 bytes = 9,000,000,000,000 bytes = 9TB

Storage conclusion: 9TB requires SHARDING
Write throughput: 200/sec fine on single server
Read throughput: 20K/sec needs multiple replicas/caches

    Your Action: ALWAYS calculate these FIRST:

    1. DAU (Daily Active Users)
    2. 1M DAU? 100M DAU? 1B DAU?

    3. Completely different scale

    4. QPS (Queries Per Second)

    5. Calculation: DAU √ó avg requests/user √∑ 86,400 seconds

    6. Makes capacity clear

    7. Storage

    8. Calculation: QPS √ó avg data size √ó retention period

    9. Determines: single server vs sharding vs archival

    10. Bandwidth

    11. Calculation: QPS √ó avg data size
    12. Determines: network, API gateway capacity

    Complete Calculations for URL Shortener:

    DAU: 1M users
Avg URLs created: 10/day
QPS: 1M √ó 10 √∑ 86,400 = 115 writes/sec
Read/write ratio: 100:1 = 11,500 reads/sec

Storage per URL:
- Short code: 7 bytes
- Long URL: 1KB
- Metadata: 100 bytes
- Total: ~1.1KB

Yearly storage: 115 writes/sec √ó 86,400 sec √ó 365 days √ó 1.1KB
= 115 √ó 86,400 √ó 365 √ó 1.1KB = 3.6TB/year

5-year retention: 18TB (needs sharding? probably not at this scale)

    Design Impact: - 115 writes/sec ‚Üí single DB server fine (can handle 1000s) - 11,500 reads/sec ‚Üí need caching + replicas - 18TB ‚Üí if using single drive, archive old data

    4Ô∏è‚É£ Missing Non-Functional Requirements

    What Most Candidates Do: Only plan functional (create, read, delete)

    What You Must Do: Also define NFRs

    Non-Functional Requirements: - Latency: How fast must response time be? - Availability/SLA: What's acceptable downtime? (99%, 99.9%, 99.99%) - Consistency Model: Strong? Eventual? Causal? - Throughput: Requests/second targets?

    Example Failure:

    Candidate: "We store URLs in database, cache in Redis"
Interviewer: "If Redis dies?"
Candidate: "Use the database..."
Interviewer: "Latency impact?"
Candidate: "It will be slow?"

Interviewer: "By how much? 1s? 10s? 1m?"
Candidate: "I don't know?"
Interviewer: ‚ùå FAIL - No latency SLA defined

    Better Approach:

    "Our latency SLA: p99 < 100ms
If Redis dies:
  - Fallback to database (slower, ~500ms)
  - Degrade: show cached version if available
  - Alert: page on-call to rebuild Redis (RTO 5 min)
  - This keeps p99 < 500ms temporarily"

    NFR Examples Per System:

    System Latency SLA Availability Consistency
    Twitter p99 < 100ms 99.9% Eventual
    Uber p99 < 50ms 99.95% Strong
    Gmail p99 < 1s 99.99% Strong
    Analytics p99 < 1m 95% Eventual

    5Ô∏è‚É£ Ignoring Failure Scenarios

    Problem: Your design works when everything is fine. Fails under chaos.

    Common Failures:

    Network Partition

    Problem: Server A can't reach Server B
Question: Timeout? Retry? Failover?

Design flaw: 
  You: "Server A waits for Server B response forever"
  ‚Üí Cascading failure

Better:
  You: "1-second timeout, after 3 retries ‚Üí failover to backup"

    Replica Lag

    Problem: Write on primary, read from stale replica
Question: How old is acceptable?

Design flaw:
  You: "Just replicate, don't worry about lag"
  ‚Üí User writes comment, reads stale feed (comment missing)

Better:
  You: "For read-after-write: read-through-primary
       For other reads: eventual consistency (< 1s lag)
       Monitor and alert if lag > threshold"

    Cascading Failures

    Problem: Service A slow ‚Üí timeouts ‚Üí Service B backed up
        ‚Üí Thread pool exhausted ‚Üí Service B fails
        ‚Üí Service C can't call B ‚Üí Service C fails

Design flaw: No circuit breakers, no bulkheads

Better:
  Circuit breaker: If B fails, C stops calling B immediately
  Bulkhead: Limit threads per downstream service
  Timeout: Short timeout so C doesn't wait forever

    Data Loss

    Problem: Hard drive fails, data lost
Question: Do you have replicas? Backups? How many?

Design flaw:
  You: "Just use the database"
  ‚Üí One server dies, all data gone

Better:
  You: "3-replica Cassandra cluster (replication factor 3)
       Nightly snapshots to S3
       RTO 5 minutes, RPO 1 hour"

    Your Mitigation Strategy (per failure type): - Timeouts: Kill slow requests - Circuit Breakers: Stop calling failed services - Fallbacks: Use cache, degraded service, backup - Monitoring: Alert on errors, latency - Redundancy: Multiple replicas, multiple datacenters

    6Ô∏è‚É£ Not Discussing Tradeoffs Explicitly

    Failure: Propose solution, move on.

    Success: State what you're sacrificing for each choice.

    Interview Example:

    You: "I'd use hash-based sharding"
Interviewer: "Tradeoffs?"
You: "Uh... none?"
Interviewer: ‚ùå FAIL - Every choice has costs

Better:
You: "I'd use hash-based sharding for even distribution.
     Trade-off: redistribution is expensive if cluster grows.
     Benefit: no hot shards in normal operation.
     Monitoring: watch for hot shards, if detected ‚Üí migrate to directory-based."

    Sharding Strategy Example:

    Option Pro Con When
    Hash-Based Distributes evenly Expensive rebalance Initial deployment
    Range-Based Easy to add servers Uneven distribution (hot range) When ranges skew predictable
    Directory-Based Handles uneven Directory becomes bottleneck When many hot shards

    Your Approach:

    Initial: Hash-based sharding
  - Assumption: even distribution
  - Monitoring: track shard sizes

If observation: "Shard 3 has 60% of data"
  - Root cause: hash function poor or data skewed
  - Decision: Migrate to directory-based
  - Justification: "Accept directory overhead for uniform distribution"

    Always State Trade-offs Like: 1. "I'm trading [cost A] for [benefit B]" 2. "Assumption: [X condition holds]" 3. "Monitoring alert: [Y metric becomes bad]" 4. "If [Y happens], migrate to: [backup plan]"

    7Ô∏è‚É£ Not Justifying Each Component

    Failure: "Load balancer, API gateway, queue, cache, database..."

    Reality: Many candidates can't justify each one

    You Must Be Able To Explain:

    WHY load balancer? (1 sentence)
WHY API gateway? (1 sentence)
WHY queue? (1 sentence)

    Correct Framework:

    Problem ‚Üí Solution ‚Üí Component (justified)

PROBLEM: 1M users, single server can handle 10K QPS max
SOLUTION: Distribute load across multiple servers
COMPONENT: Load balancer ‚úÖ
JUSTIFICATION: "Routes requests to available servers, handles failover"


PROBLEM: Different services (auth, tweets, feed) need different scales
SOLUTION: Rate limit & authenticate at entry point
COMPONENT: API gateway ‚úÖ
JUSTIFICATION: "Centralized auth, rate limiting, request routing"


PROBLEM: Write spike (1M likes/sec) causes database lock
SOLUTION: Buffer writes, process asynchronously
COMPONENT: Queue (Kafka, RabbitMQ) ‚úÖ
JUSTIFICATION: "Decouples write spike from database, allows batch processing"


PROBLEM: Database queries take 500ms, users expect < 100ms
SOLUTION: Store frequently accessed data in fast memory
COMPONENT: Redis cache ‚úÖ
JUSTIFICATION: "< 5ms memory access vs 500ms disk access"


PROBLEM: Need durability & consistency for transactions
SOLUTION: Structured, ACID-compliant storage
COMPONENT: Primary database (PostgreSQL) ‚úÖ
JUSTIFICATION: "Relational data with ACID guarantees"

    Anti-Pattern:

    ‚ùå "We'll use Kafka because it's cool"
‚ùå "Redis because Netflix uses it"
‚ùå "Mongo because no Schema"

‚úÖ "Kafka because we need to decouple writes at scale"
‚úÖ "Redis because sub-100ms reads critical"
‚úÖ "Mongo because schema evolves independently per service"

    System Design Study Strategy

    The Study Flow

    1. Study Existing Solutions (Grokking: 15h)
    2. Understand patterns, not memorize

    3. Focus on WHY, not HOW

    4. Understand Theory (DDIA: 10h)

    5. Read chapters on weak areas

    6. Replication, partitioning, consistency

    7. Reverse Engineer Real Systems (5h)

    8. Design Twitter without reading solutions
    9. Design Uber from scratch

    10. Compare to real implementations

    11. Mock Interview (ongoing)

    12. Practice talking through design
    13. Justify every choice
    14. Document tradeoffs
