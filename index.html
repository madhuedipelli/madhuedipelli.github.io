    ‚ö†Ô∏è The 7 DSA Gotchas to Avoid

    1. The Magic Bullet Trap ‚≠ê MOST COMMON

    Problem: Jumping straight to optimal solution, failing when wrong, wasting 20+ minutes.

    Fix: Always start naive ‚Üí optimize incrementally ‚Üí test baseline.
    Your Takeaway: In interview, propose naive first, get feedback, then optimize.

    2. Not Testing on Paper

    Problem: Algorithm works in your head but fails in reality.

    Fix: Trace through 2-3 examples on paper BEFORE coding.
    Real Example: Reversing linked list [4,5,1,3,2] - works in head, fails on paper.

    3. Speed ‚â† Understanding ‚≠ê CRITICAL

    Problem: L4 interviews need 15min, L5 needs 3-5min. If not autonomous, redo the topic.

    Benchmark: - L3: 30+ minutes - L4: 15 minutes
    - L5: 3-5 minutes

    Fix: Track time per problem, watch for improvement curve.

    4. Only Memorizing Solutions

    Problem: Watch solution, move on. Can't apply to variations.

    Fix: For EACH solution, reverse-engineer: - What problem characteristics led here? - What inefficiencies existed in naive? - What insights bridge naive ‚Üí optimal?

    Outcome: 20 deeply understood problems > 100 memorized problems

    5. Pattern Confusion Overlap

    Problem: After learning many patterns, preconditions overlap and confuse you.

    Fix: After learning new pattern, backtest against previous 5 problems.
    Timeline: Spend 30 min per new pattern on overlap checking.

    6. The Sliding Window Trap

    Problem: Memorize "use sliding window for subarrays" without understanding WHY.

    Fix: Learn the PRECONDITION: "Can reuse portion of previous result"

    7. Skipping Edge Cases in Planning

    Problem: Code first, realize edge cases after. Wastes time, causes bugs.

    Fix: BEFORE coding, list all edge cases: - Empty input, single element, duplicates, negatives - Extreme values, already sorted/reverse sorted

    ‚ö†Ô∏è The 7 System Design Gotchas to Avoid

    1. Premature Component Specification ‚≠ê MOST COMMON

    Failure Pattern:

    ‚ùå "We'll use MongoDB, Redis, Nginx..."
Interview: "Why?"
Your Answer: "Because... it scales?"
Interviewer: ‚ùå FAIL

    Fix: Start with PROBLEM characteristics, then JUSTIFY tech choice.

    Correct Pattern:

    ‚úÖ "We need to store non-relational data with horizontal scaling.
   MongoDB provides this by sacrificing consistency for availability."

    2. The Free Lunch Fallacy

    Myth: "NoSQL scales better"

    Truth: NoSQL trades consistency/joins for scale - no magic.

    Tradeoff Matrix:

    Aspect MySQL NoSQL
    Joins ‚úÖ Fast ‚ùå Complex
    Scale ‚ö†Ô∏è With sharding ‚úÖ Native
    Consistency ‚úÖ Strong ‚ùå Eventual
    Simplicity ‚úÖ ‚ùå

    Your Takeaway: EVERY choice costs something. Name the cost.

    3. Not Calculating Capacity ‚≠ê CAUSES WRONG DESIGN

    Example Calculations:

    500M URLs/month √∑ 30 days √∑ 86400 sec = 200 writes/sec
100:1 read/write ratio = 20,000 reads/sec
Each entry ~500 bytes
3 years retention = 18B URLs √ó 500B = 9TB

    Insight: 9TB = needs sharding. 200 writes = probably fine on single server IF no sharding

    Your Action: ALWAYS calculate: DAU, QPS, storage, bandwidth FIRST

    4. Missing Non-Functional Requirements

    What Most Candidates Do: Only plan functional (create, read, delete)

    What You Must Do: Also define: - Latency requirement (milliseconds?) - Availability/SLA (99.9%? 99.99%?) - Consistency model (strong? eventual?) - Throughput needed (requests/sec?)

    Example Failure:

    Candidate: "We store URLs in database, cache in Redis"
Interviewer: "If Redis dies?"
Candidate: "Use the database..."
Interviewer: "Latency impact?"
Candidate: "Goes down?"

Better:
"If Redis dies, fallback to database (slower)
We set 1-second timeout so app doesn't hang
We alert on Redis failure to rebuild quickly"

    5. Ignoring Failure Scenarios

    Failure Types: - Network: Server A can't reach Server B ‚Üí Timeout? Retry how many times? - Replica Lag: Write on primary, read stale replica ‚Üí How old acceptable? - Cascading: Service A slow ‚Üí timeouts ‚Üí Service B backed up - Data: Disk fails ‚Üí Need replication

    Your Mitigation Strategy: Timeouts, circuit breakers, fallbacks, monitoring

    6. Not Discussing Tradeoffs Explicitly

    Failure: Propose solution, move on.

    Success: State what you're sacrificing for each choice.

    Example - Sharding Strategy:

    Option Pro Con
    Hash-Based Even distribution Redistribution nightmare
    Range-Based Easy to add servers Potentially uneven
    Directory-Based Handles uneven Directory becomes bottleneck

    Right Answer:

    "I'd use hash-based sharding initially for even distribution
(acknowledging risk of uneven load in production)
With monitoring to detect hot shards
If observed, migrate to directory-based despite complexity"

    7. Not Justifying Each Component

    Failure: "Load balancer, API gateway, queue, cache, database..."

    Success: Can you explain WHY each component exists in one sentence?

    Correct Framework:

    Problem ‚Üí Solution ‚Üí Component (justified)

Problem: 1M requests, single server can't handle
Solution: Multiple servers + distribute requests
Component: Load balancer ‚úÖ (justified)

Problem: Latency from database lookups
Solution: Store frequent queries in fast memory
Component: Redis cache ‚úÖ (justified)

Problem: Spike traffic causes failures
Solution: Buffer requests, process asynchronously  
Component: Message queue ‚úÖ (justified)

    ‚úÖ Interview Readiness Checklist

    DSA: Are you ready to apply?

    • [ ] Completed 60%+ of Grokking
    • [ ] Completed EPI chapters 5-16
    • [ ] Can solve 70% of 20 random problems first try
    • [ ] Can solve ANY problem with infinite time
    • [ ] Only mistakes are weird situational facts, not core patterns
    • [ ] Practice diary filled with 50+ problems
    • [ ] Can speed-run 5 problems in 45-60 min total

    System Design: Are you ready?

    • [ ] Studied 5-10 case studies deeply (Grokking)
    • [ ] Read DDIA chapters on your weak areas
    • [ ] Can explain WHY each system design choice made
    • [ ] Can calculate: DAU, QPS, storage, bandwidth
    • [ ] Can identify failure scenarios and mitigations
    • [ ] Can state tradeoffs explicitly
    • [ ] Never say tech name without justifying

    Mindset Shifts Required

    • [ ] Shift from "memorization" to "understanding"
    • [ ] Shift from "speed" to "consistency"
    • [ ] Shift from "complexity" to "simplicity"
    • [ ] Shift from "patterns" to "preconditions"
    • [ ] Shift from "tech stacks" to "reasoning"
