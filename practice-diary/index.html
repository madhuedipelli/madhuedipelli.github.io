    The Practice Diary Template

    Create one markdown file per problem. Example: 2025-02-15-two-sum.md

    # Two Sum

**Date:** February 15, 2026  
**Time Spent:** 28 minutes  
**Result:** ‚úÖ Passed / ‚ö†Ô∏è Partial / ‚ùå Failed  
**Attempt:** 1st attempt / 2nd attempt / 3rd attempt  

## Problem Statement
Two integers in array sum to target, return indices

## Approach
- Naive: O(n¬≤) brute force
- Optimal: O(n) hash map - single pass

## Solution

### Brute Force (Baseline)
```python
def twoSum(nums, target):
    for i in range(len(nums)):
        for j in range(i+1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []

    Complexity: O(n¬≤) time, O(1) space

    Optimal (Hash Map)

    def twoSum(nums, target):
    seen = {}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []

    Complexity: O(n) time, O(n) space

    Mistakes Made

    Mistake 1: Off-by-One Index

    What I Did: Returned [i, j] after loop, wrong indices
    Why: Forgot to store index while building map
    Cost: 5 minutes debugging
    Prevention: "Always store index IN the dictionary iteration"

    Mistake 2: Didn't Check Edge Case

    What I Did: Assumed array always has solution
    Why: Problem didn't explicitly say "always has answer"
    Cost: Would fail on empty array
    Prevention: "Read problem for edge cases BEFORE coding"

    Decision Chain

    Start ‚Üí "Two numbers sum to target?"
     ‚Üí "Naive: brute force O(n¬≤)" ‚úì Understood
     ‚Üí "Optimal: speed up lookup?" 
     ‚Üí "Use hash map for O(1) lookup" ‚úì Right
     ‚Üí "Store num‚ûúindex mapping" ‚úì Right
     ‚Üí "Check complement exists BEFORE storing" ‚ö†Ô∏è Tricky
     ‚Üí "Finally: handle no solution edge case" ‚úì Right

    Key Decision Point: Check complement BEFORE storing (order matters!)

    Testing Trace

    Input: [2, 7, 11, 15], target = 9

Iteration 1: num=2, complement=7, seen={}, store: seen[2]=0
Iteration 2: num=7, complement=2, seen[2] exists! Return [0, 1] ‚úì

Input: [1,1,1], target = 3
Iteration 1: num=1, complement=2, seen={}, store: seen[1]=0
Iteration 2: num=1, complement=2, seen={1}, store: seen[1]=1
Iteration 3: num=1, complement=2, seen={1}, store: seen[1]=2
Return [] ‚úì Edge case handled

    Key Insights

    1. Precondition for one-pass hash map: Need to store value‚Üíindex, check OTHER value existed previously

    2. Why this matters: Two-pass (build map, then search) is safer but slower

    3. Pattern recognition: Any problem where "lookup value" = "search previous results" ‚Üí hash map works

    Similar Problems

    Problem Similarity Why Different
    3Sum Similar: find targets Harder: 3 values, still use hashes
    Two Sum II Similar: find pair Different: array pre-sorted (use two pointers!)
    Closest Sum Similar: hash map Harder: need closest, not exact

    Related Follow-ups

    Interviewer might ask: 1. "What if we need ALL pairs?" ‚Üí O(n) still works, store list 2. "What if there are duplicates?" ‚Üí Edge case traced ‚úì 3. "What if array is sorted?" ‚Üí Could use two-pointers O(n) but use hashes OK 4. "What about space constraints?" ‚Üí Trade O(n) space vs O(n¬≤) time

    Time Analysis

    • Thinking: 2 min (understood problem)
    • Naive: 3 min (wrote brute force)
    • Optimal: 5 min (thought about hash map)
    • Coding: 12 min (wrote solution)
    • Testing: 6 min (traced examples)
    • Total: 28 min

    Speed Assessment: - 28 min = slow for easy problem (target 5-10 min) - After 20h practice: should be 5 min - After 50h practice: should be 2 min - Track this to ensure improvement

    Lessons for Future

    ‚úÖ Do: - Hash map for "lookup" problems - Store metadata (indices) not just keys - Check edge cases on paper first - Trace through examples methodically

    ‚ùå Avoid: - Assuming array always has solution - Off-by-one errors in index storage - Coding without tracing first

## How to Use the Template

### For Every Problem:

1. **Date & Time:** Track when you solved it, how long
2. **Problem Statement:** 1-line summary
3. **Approach:** Naive AND optimal (always compare)
4. **Solution:** Code both, with complexity analysis
5. **Mistakes:** THIS IS KEY
   - What did you do wrong?
   - WHY did you do it? (root cause)
   - How much time did it cost?
   - How will you prevent next time?
6. **Decision Chain:** Show your thinking path
7. **Testing:** Trace on paper for 2-3 examples
8. **Key Insights:** What pattern/precondition unlocked this?
9. **Similar Problems:** Link to related problems
10. **Follow-ups:** What would interviewer likely ask?

### Optional But Helpful:

- **Comparison to Reference:** How did your solution compare?
- **Lessons:** General takeaways for future problems
- **Confidence Level:** 1-10 how confident you are in this solution
- **Repeat Date:** If you solve this again, note the date and compare

    Common Patterns to Track

    As you go through problems, note patterns in your diary:

    Pattern: Hash Map for Lookup

    When to use:
- "Find X in array where property Y"
- "Complement lookup"
- "Count frequencies"

Key insight:
- Store key‚ûúvalue while iterating
- Lookup = previous element met condition

    Pattern: Sliding Window

    When to use:
- Subarray/substring problems
- When can reuse portion of result

Precondition:
- "Are we adding/removing ends?"
- "Is previous result relevant?"

    Pattern: Two Pointers

    When to use:
- Sorted array
- Find pair with constraint
- Container with most water

Precondition:
- Array must be sorted
- Working from both ends
